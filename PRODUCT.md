# Product

## Register

product

## Users

Hunky is for Mac users who work with CD-based game dumps and CHD archives but do not want to install MAME, Homebrew, or run terminal commands. They may be archivists, emulator users, preservation hobbyists, or technically curious collectors. Their context is often batch-oriented: they are checking a folder of disc images, confirming references and integrity, then letting a queue run without babysitting every file.

## Product Purpose

Hunky is a self-contained macOS front end for MAME's `chdman`. It should make CHD creation, extraction, inspection, and verification feel safe, local, and understandable. Success means users can drop files or folders, trust the proposed action, see integrity issues before work starts, and never wonder whether Hunky overwrote or deleted their data.

## Brand Personality

Hunky is a Retro Console Workbench: playful, tactile, and game-adjacent, but still careful with user files. It should feel like a late optical-console service bay on a Mac: disc trays, memory-card save paths, BIOS ready checks, status LEDs, and textured graphite plastic. The product can be fun without turning warnings, output paths, or destructive boundaries into jokes.

## Anti-references

Avoid brand-infringing console logos, arcade casino neon, generic sci-fi dashboards, terminal cosplay, sepia archive nostalgia, oversized marketing hero layouts, glassy custom controls, decorative motion without state, and illegible novelty text. Do not make the app feel like a game launcher; Hunky is a tool for game preservation work.

## Design Principles

- Make safety visible: output location, collision behavior, audit results, and destructive boundaries should be plain.
- Let the queue be the product: the main screen should optimize scanning, comparison, and repeated action.
- Use optical-console cues as affordances: disc bay, queue slots, memory-card save path, BIOS ready check, and status LEDs.
- Keep core commands literal: Create CHD, Extract, Info, Verify, Retry, Remove, Run, Stop.
- Reserve bright color for state and action: cyan for start/running/progress, green for success, amber for caution, red for critical, violet for confirmed Redump.
- Keep domain detail close to the work: file type, platform, Redump status, CRC, and output actions belong in rows and panels, not in decorative framing.

## Accessibility & Inclusion

Target at least the native macOS accessibility baseline: keyboard operation, visible focus, VoiceOver labels for icon-only controls, readable contrast in dark mode, and reduced-motion support for all nonessential animation. Do not rely on color alone for Redump, warning, error, or running states. Decorative console texture and generated art must never be required to understand the UI.
